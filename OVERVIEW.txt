╔═════════════════════════════════════════════════════════════════════════════╗
║  1. WHAT IT DOES                                                            ║
╚═════════════════════════════════════════════════════════════════════════════╝

An automated MLB betting bot trading event contracts on Kalshi's KXMLBGAME
moneyline market. On each MLB game day it:

  1. Fetches vig-free probability blends from sharp bookmakers (Pinnacle,
     LowVig, BetOnline) via The Odds API.
  2. Runs an XGBoost model as a secondary signal — used to confirm or veto
     each Pinnacle-sourced bet, plus as a standalone fallback when no sharp
     line is available.
  3. Checks Kalshi's orderbook for each game, computes net-of-fee expected
     value (EV) on both sides at the current ask.
  4. Places YES buy orders when edge clears a dynamic threshold, sized as a
     flat dollar unit per bet (no Kelly).
  5. Stays alive in watch mode, amending resting orders if prices drift and
     cancelling them if EV turns negative.
  6. Settles outstanding bets against Kalshi (cross-checked vs. MLB API) the
     following morning, logging realized PnL to the ledger.

Kalshi is the ONLY venue. Only moneyline (KXMLBGAME) is supported — Kalshi
doesn't list run line / totals / F5 / props for MLB. Sizing is a flat $10
per qualifying bet, sized to the current ~$30 bankroll for a data-collection
phase under the new 0.5% EV gate.


╔═════════════════════════════════════════════════════════════════════════════╗
║  2. DESIGN PHILOSOPHY                                                       ║
╚═════════════════════════════════════════════════════════════════════════════╝

PINNACLE-PRIMARY, MODEL-FALLBACK-ONLY.

The core architectural choice: Pinnacle's blended sharp probability is the
primary betting signal. As of 2026-05-01, the XGBoost model has NO veto power
on Pinnacle-sourced bet selection. It serves only two roles:
  - Fallback to model-only when Pinnacle has no line (with stricter gates)
  - Circuit-breaker on catastrophic model failure (gap > 50pp), as a data-
    quality sanity check, NOT a forecast-quality gate

The 2024-2025 model-vs-Pinnacle CLV backtest (analysis/model_vs_pinnacle_clv.py
and analysis/inversion_check.py) showed the model is GLOBALLY well-calibrated
but LOCALLY anti-calibrated specifically on disagreement games. When the model
and the Pinnacle blend pick OPPOSITE sides:
  - At ≥10pp gap: blend wins 59% of head-to-head
  - At ≥15pp gap: blend wins 64%
  - At ≥20pp gap: blend wins 77%
  - At ≥25pp gap: blend wins 85%

The original 20pp gap-veto and the directional model_p ≥ ask veto were
rejecting exactly the games where Pinnacle is most reliable. Removing them
corrects a known false-veto failure mode. Expected ROI uplift is unknown
until forward-going daily_stats data accumulates (target: 60 days) — we
removed false vetoes but haven't established that Kalshi actually misprices
the games we'll now be eligible to bet on.

This bounds theoretical upside (we can't beat Pinnacle's closing line) but
maximizes robustness. Considered and rejected: trusting the model more. The
model's features (team Pythagorean, Log5, SP rolling stats) are strictly a
subset of what Pinnacle incorporates. Trusting the model to overrule Pinnacle
is a long-run losing strategy — empirically verified at all gap levels.

KALSHI IS TIGHT ON MLB MONEYLINES. Empirically, Kalshi's ask typically sits
1–2¢ above Pinnacle fair. Add the quadratic fee (~1.5¢ effective) and the
distribution of best_ev across the slate clusters tightly around 0%, with
most games landing between -3% and 0%. Under the original 3% gate, 0–1 bets
per slate was the norm.

THIN-EDGE PHASE (2026-04-27 →). The EV gate was lowered from 2.5% (dynamic
floor) to 0.5% to start collecting actual placed-bet data on the gap↔EV
correlation hypothesis: that high model-vs-market gap rows tend to also be
the highest-EV ones. Individual 0.5% bets are nearly coin flips on expected
value (calibration noise dominates the displayed edge), so this phase is
explicitly a data-collection bet, not a high-confidence growth strategy.
Expect ~1 bet per slate at this gate, $10 each, at $30 bankroll = three
losses in a row would zero the bankroll — risk is real and acknowledged.

╔═════════════════════════════════════════════════════════════════════════════╗
║  4. THE MODEL                                                               ║
╚═════════════════════════════════════════════════════════════════════════════╝

ARCHITECTURE: XGBoost binary classifier, calibrated via isotonic regression
(CalibratedClassifierCV with 5-fold CV). Predicts P(home team wins).

TRAINING DATA: 2015–2025 regular-season games from MLB Stats API. 24,686
total games, 23,817 after 5-game-minimum filter. 5-fold stratified CV, then
walk-forward test on held-out final year, then final fit on full data.

FEATURES (53 total in current model):

  Team (20):
    home/away YTD:    wins, losses, win_pct, pythagorean
    home/away L5:     wins, losses, win_pct, pythagorean
    combined:         pyth_delta, log5, r5_pyth_delta, r5_log5

  SP — home/away L5 (12):
    era, whip, k_pct, bb_pct, k_bb_pct, hr_per_9

  SP — home/away YTD (8):
    era, whip, k_pct, bb_pct

  SP — context (6):
    days_rest, starts_ytd, is_prior (1=rookie/early-season; league-avg fallback)

  SP — matchup deltas (7):
    l5_era_delta, l5_whip_delta, l5_k_pct_delta, l5_bb_pct_delta,
    l5_k_bb_pct_delta, ytd_era_delta, ytd_k_pct_delta

  Note: home_r5_win_pct dropped to 0.0000 importance — collinear with
  wins/losses. away_sp_is_prior also 0.0000 — XGBoost captured the rookie
  signal via home_sp_is_prior alone. Harmless; no pruning needed yet.

LEAKAGE DISCIPLINE: rolling stats computed BEFORE each game only. For game i,
all inputs derive from games 0..i-1. Prior-fallback (league average) used when
fewer than 3 prior starts exist for a pitcher — flagged via sp_is_prior.

CURRENT PERFORMANCE (held-out 2025 walk-forward):
  Overall CV: 56.0% ± 0.5% accuracy, logloss 0.685
  ≥60% conf: 56.6% acc (n=799)
  ≥63% conf: 60.0% acc (n=512)
  ≥66% conf: 65.9% acc (n=323)

KEY INSIGHT: team features still dominate (log5 + pyth_delta = top 2). SP
features added real but modest lift — ~1.5pp at high confidence levels.
The sp_is_prior flag specifically fixed the MIN@NYM-type case where a rookie
home starter's 4.30-ERA prior was getting compared to a real 3.29 away
starter, falsely inflating away-team prediction.

LIMITATIONS (known, not yet addressed):
  - No park factors
  - No weather
  - No bullpen state
  - No umpire tendencies
  - No sp_small_sample flag for 3–5 start pitchers with extreme early numbers
    (caused NYY@BOS 12.7% gap from Schlittler's 1.95 ERA on ~3 starts)


╔═════════════════════════════════════════════════════════════════════════════╗
║  5. STAKING MATH                                                            ║
╚═════════════════════════════════════════════════════════════════════════════╝

EV is computed NET of Kalshi's quadratic taker fee. Sizing is flat-unit —
every bet that passes the gate is the same dollar amount, regardless of edge.

FEE FORMULA (empirically verified against real settlements):
  per-contract fee = 0.07 × fee_multiplier × P × (1 - P)

  fee_multiplier is pulled live from GET /series/KXMLBGAME at each run.
  MLB markets use multiplier = 1.0. Fallback to 0.07 if the request fails.

NET EV:
  b_gross = (1 - price) / price           # gross decimal odds from Kalshi price
  fee_rate_on_winnings = 0.07 × multiplier × price
  b_net = b_gross × (1 - fee_rate_on_winnings)
  EV = p_model × b_net − (1 − p_model)

FLAT-UNIT SIZING:
  contracts = floor(FLAT_UNIT_DOLLARS / ask_price)
  cost      = contracts × ask_price
  (Returned by risk.flat_unit_contracts(); see also gates below.)

  FLAT_UNIT_DOLLARS = 10.00.
  Rationale for switching from 1/8 Kelly → flat unit (2026-04-27):
    - At low bankroll ($30), 1/8 Kelly stake on a 0.5% edge rounds to <$1 →
      MIN_BET_DOLLARS gate fires, bot never actually bets.
    - Flat unit is simpler to reason about; every passing bet is a known
      $10 commitment. Useful for a data-collection phase.
    - Trade-off: per-bet exposure jumped from ≤10% of bankroll (Kelly cap)
      to 33% of bankroll. Three consecutive losses ≈ ruin. Expected to be
      revisited once bankroll grows or once the gap↔EV correlation is
      validated/refuted.

ACTIVE GATES (applied around the unit):
  MIN_BET_DOLLARS = 1.00   — sanity floor; effectively dead at $10 unit
  available_cash           — bankroll − this-session resting cost; can't
                             over-spend across multiple games per session
  depth_cap                = Kalshi top-of-book depth for chosen side
                             (prevents walking the book 3–5¢ past quoted ask)

REMOVED 2026-04-27 (no longer in code):
  KELLY_FRACTION, MAX_BET_FRACTION, MAX_SESSION_EXPOSURE_FRACTION,
  risk.kelly_contracts(), risk.check_session_exposure().


╔═════════════════════════════════════════════════════════════════════════════╗
║  6. GATES & THRESHOLDS                                                      ║
╚═════════════════════════════════════════════════════════════════════════════╝

DYNAMIC EV THRESHOLD (for Pinnacle-primary bets, ≥2 sharp books):
  threshold = clamp(0.005 + 1.5 × prob_std, 0.005, 0.08)

  When sharp books agree (low std), threshold falls toward the 0.5% floor.
  When they disagree (high std), threshold rises up to an 8% ceiling. This
  is in-flight calibration of confidence in the blend.

  (Floor was 2.5% prior to 2026-04-27; lowered to 0.5% to start collecting
  placed-bet data on thin edges. The dynamic ceiling response is unchanged.)

STATIC THRESHOLDS:
  PINNACLE_MIN_EV       = 0.005  — single-sharp-book path (only 1 book quoted)
                                   (was 0.03 before 2026-04-27)
  MODEL_MIN_EV          = 0.08   — model fallback when no Pinnacle at all
                                   (kept strict; respects Pinnacle-primary
                                   hierarchy when only model signal available)
  CONFIDENCE_THRESHOLD  = 0.55   — model-only: stronger side must clear this

MODEL-PINNACLE GAP CIRCUIT BREAKER:
  MODEL_PINNACLE_GAP_MAX = 0.50

  If |p_model_home − p_pinnacle_home| > 50%, skip the game. This is a data-
  quality sanity threshold for catastrophic model failure (e.g., feature
  pipeline bug producing wild predictions), NOT a forecast-quality gate.
  Set far outside any historical legitimate disagreement — observed gaps in
  2024-2025 backfill data top out around 38pp, with reliable Pinnacle wins
  through that range (see analysis/inversion_check.py for the H2H accuracy
  curve by gap bucket). Was 0.20 prior to 2026-05-01.

(REMOVED 2026-05-01) MODEL VETO ON PINNACLE BETS:
  Previously required model_p for the bet side to be ≥ Kalshi ask. Removed
  because the CLV backtest showed the model is anti-calibrated specifically
  on disagreement games — when model and blend picked opposite sides at
  ≥10pp gap, blend won 59-85% of head-to-head depending on gap size. The
  veto was rejecting the bot's most reliable Pinnacle bets.

GATES IN ORDER, PER GAME:
  1. already have position             → skip (reason=already-positioned)
  2. team stats missing                → skip (reason=missing-stats)
  3. model-Pinnacle gap > 50% breaker  → skip (reason=model-pin-gap)
  4. no Pinnacle + confidence < 55%    → skip (reason=confidence-gate)
  5. no Kalshi orderbook               → skip (reason=no-orderbook)
  6. no side clears EV threshold       → skip (reason=no-edge)
  7. available_cash < unit floor       → skip (reason=insufficient-cash)
  8. place_order returned falsy        → skip (reason=place-failed)
  9. otherwise                         → placed (reason=placed)

(Gates 8 "session-cap" and 9 "kelly-zero" from the prior version were
removed with the Kelly system on 2026-04-27. Insufficient-cash now subsumes
both — it fires when this-session resting orders have exhausted bankroll
below the next $10 unit.)

(Gate 7 "model_p < ask" veto removed 2026-05-01 — see above.)


╔═════════════════════════════════════════════════════════════════════════════╗
║  7. WATCH MODE                                                              ║
╚═════════════════════════════════════════════════════════════════════════════╝

After placement (if --watch flag set, or cron equivalent), main.py enters
watcher.watch_and_amend(), which polls every 3 min (WATCH_INTERVAL_SECONDS)
until all games on active orders have started.

Per active order:
  1. Query Kalshi for order status; drop from active list if not 'resting'.
  2. Refresh Pinnacle blend every 10 min (to reduce API calls).
  3. Recompute EV at current ask using current Pinnacle prob (or stored p_used
     if the game fell off the Pinnacle feed).
  4. If EV < per-order ev_threshold (the dynamic threshold that admitted the
     bet at placement), CANCEL the order.
  5. If ask moved ≥ 2¢ (WATCH_AMEND_TICK_THRESHOLD) in either direction,
     AMEND the resting price to follow the market.

PER-ORDER THRESHOLD MEMORY: the dynamic EV threshold that admitted each bet
is stored on the active_orders entry. Watch mode compares against THAT, not
a fresh recomputation — otherwise an order placed at a permissive 2.5%
threshold could be unfairly cancelled by a stricter live recomputation, and
vice versa.


╔═════════════════════════════════════════════════════════════════════════════╗
║  8. CLV TRACKING                                                            ║
╚═════════════════════════════════════════════════════════════════════════════╝

Closing Line Value measures "did the market move toward our pick?" It's the
leading indicator of real edge; positive PnL with zero CLV means hot streak,
not skill. Target: +1% rolling mean confirms edge at n≈300; +2% at n≈150.

CAPTURE (standalone cron every 5 min, 11 AM–midnight ET):
  clv_snapshot.py scans ledger.csv for today-dated unsettled bets. For each,
  looks up the (away_code, home_code) match in the current sharp-book blend.
  If the game is still in the feed (pre-first-pitch), overwrites the row's
  CLV columns. Once the game locks (disappears from the feed), no further
  snapshots fire — the last-written value becomes the de-facto close.

DATE FILTER: only today-ET rows are considered. Past-dated rows missing CLV
are logged but skipped — historical closing lines aren't retrievable from
The Odds API free tier.

METRICS:
  clv_prob = p_close_fair_our_side − p_used_at_bet
             (probability-space line movement; +ve = market moved toward us)

  clv_pct  = (1/bet_price) / (1/p_close_fair_our_side) − 1
             (ROI-space; our entry odds vs market close)

  For Pinnacle-sourced bets, p_used IS the vig-free blend at bet time, so
  clv_prob measures clean market-movement CLV. For model-sourced bets,
  p_used is the model prob; clv_prob then measures "did the market agree
  with what my model predicted."

REPORTING: settle.py's print_summary now includes a CLV SUMMARY section:
  - Mean clv_prob, clv_pct
  - Rolling-100 mean clv_prob
  - Breakdown by prob_src (Pinnacle / Blend / model)
  - Breakdown by edge-bucket (0–3%, 3–4%, ..., 6%+)


╔═════════════════════════════════════════════════════════════════════════════╗
║  9. SETTLEMENT                                                              ║
╚═════════════════════════════════════════════════════════════════════════════╝

settle.py runs every morning at 2 AM ET via cron. For each unsettled ledger
row:

  1. Fetch Kalshi's /portfolio/settlements for the row's ticker. Skip if
     Kalshi hasn't settled yet.
  2. Parse market_result: 'yes' / 'no' / 'void' / 'scalar' (unexpected).
  3. Use Kalshi's reported fee_cost (actual, not computed) for precise PnL:
     - win: gross = contracts × (1 − price); pnl = gross − fee_cost
     - loss: pnl = -contracts × price
     - void: pnl = -fee_cost (usually $0)
  4. Cross-check against MLB Stats API's Final winner for that game date.
     Log any divergence but trust Kalshi's result.
  5. Call ledger.record_outcome() to close the row.

After settlement pass, print_summary() renders:
  - Wins/losses/voids
  - Win rate (excluding voids)
  - Wagered, net PnL, ROI
  - By-date breakdown
  - CLV summary (as detailed above)

24 columns. Auto-migrates when FIELDNAMES changes — legacy rows get new

  game_date            — YYYY-MM-DD
